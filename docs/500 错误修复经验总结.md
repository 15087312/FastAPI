## 🔍 错误案例汇总

### 案例 1：`'Lock' object has no attribute '__enter__'`

#### 错误现象
```json
{
  "success": false,
  "message": "'Lock' object has no attribute '__enter__'"
}
```

**HTTP 状态码**: 500  
**影响接口**: `/api/v1/inventory/adjust` 等所有使用分布式锁的接口

#### 根本原因
1. **Redlock 库版本差异**: `redlock-ng` 2.0+ 版本的 `lock()` 方法返回的是 `LockContext` 对象，而非直接的锁对象
2. **上下文管理器调用失败**: `LockContext.__enter__()` 方法在某些情况下会抛出异常
3. **异常处理缺失**: `RedLockAdapter.lock()` 方法没有捕获 `__enter__()` 调用时的异常

#### 技术细节
```python
# 问题代码（旧版本）
def lock(self, resource, ttl=None):
    lock_context = self._redlock.lock(resource, ttl)
    if lock_context:
        lock_obj = lock_context.__enter__()  # ❌ 可能抛出异常
        if lock_obj:
            return (lock_context, lock_obj)
    return None

# 修复代码（新版本）
def lock(self, resource, ttl=None):
    try:
        lock_context = self._redlock.lock(resource, ttl)
        if lock_context:
            lock_obj = lock_context.__enter__()
            if lock_obj:
                return (lock_context, lock_obj)
    except Exception as e:
        # ✅ 捕获异常，返回 None，避免 500 错误
        return None
    return None
```

#### 修复方案
**文件**: `app/core/redis.py`  
**修改位置**: `RedLockAdapter.lock()` 方法  

**关键改动**:
- 添加 `try-except` 异常捕获块
- 当 `__enter__()` 失败时返回 `None` 而不是抛出异常
- 保持与现有 `_acquire_lock()` 逻辑的兼容性

#### 验证步骤
1. 修改代码后重新构建 Docker 镜像
   ```bash
   docker-compose -f docker-compose.prod.yml up -d --build --no-deps app
   ```

2. 查看日志确认无报错
   ```bash
   docker-compose -f docker-compose.prod.yml logs -f app
   ```

3. 测试 API 接口
   ```powershell
   $body = @{warehouse_id="WH01"; product_id=220; adjust_type="increase"; quantity=10; reason="测试"; operator="admin"} | ConvertTo-Json
   Invoke-RestMethod -Uri "http://localhost:8000/api/v1/inventory/adjust" -Method Post -Body $body -ContentType "application/json; charset=utf-8"
   ```

### 案例 2：外键约束违反 `ForeignKeyViolation`

#### 错误现象
```json
{
  "success": false,
  "message": "(psycopg2.errors.ForeignKeyViolation) insert or update on table \"product_stocks\" violates foreign key constraint \"product_stocks_product_id_fkey\"\nDETAIL: Key (product_id)=(1) is not present in table \"products\"."
}
```

**HTTP 状态码**: 500  
**影响接口**: 所有涉及商品 ID 的库存操作接口

#### 根本原因
1. **数据不一致**: 尝试对不存在的商品进行库存操作
2. **外键约束**: `product_stocks.product_id` 字段有外键约束，必须引用 `products.id` 中存在的值
3. **测试数据过期**: 数据库中商品 ID 发生变化（从 1 变为 220），但 API 调用仍使用旧 ID

#### 技术细节
```sql
-- 数据库表结构
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    sku VARCHAR(64) UNIQUE NOT NULL,
    name VARCHAR(255) NOT NULL
);

CREATE TABLE product_stocks (
    id SERIAL PRIMARY KEY,
    warehouse_id VARCHAR(32) NOT NULL,
    product_id INTEGER NOT NULL REFERENCES products(id),  -- ⚠️ 外键约束
    available_stock INTEGER DEFAULT 0,
    ...
);
```

#### 解决方案

**方案 1: 查询现有商品 ID**
```bash
# 执行 SQL 查询获取当前商品 ID
docker-compose -f docker-compose.prod.yml exec db psql -U postgres -d mydb -c "SELECT id, sku, name FROM products;"
```

输出示例:
```
 id  |     sku      |   name
-----+--------------+----------
 220 | TEST_REL_007 | 测试商品
```

**方案 2: 创建测试商品**
```sql
-- 如果数据库中没有商品，先创建
INSERT INTO products (sku, name) VALUES ('TEST_SKU_001', '测试商品');
```

**方案 3: 在应用层添加校验**
```python
# 建议改进：在执行库存操作前验证商品是否存在
from sqlalchemy import select

def validate_product_exists(db: Session, product_id: int):
    """验证商品是否存在"""
    product = db.execute(
        select(Product).where(Product.id == product_id)
    ).scalar_one_or_none()
    
    if not product:
        raise HTTPException(
            status_code=404,
            detail=f"商品 ID {product_id} 不存在"
        )
```

#### 最佳实践
1. **先查询后操作**: 在执行库存操作前，先查询可用的商品 ID
2. **添加友好提示**: 返回清晰的错误信息，指导用户使用正确的商品 ID
3. **数据初始化**: 在数据库迁移脚本中添加基础测试数据

## 💡 总结

### 核心教训
1. **异常处理至关重要**: 任何外部依赖调用都应该包裹在 try-except 中
2. **数据一致性是基础**: 确保测试数据和生产数据的准确性
3. **日志是最好的老师**: 详细的日志可以快速定位问题根因
4. **并发控制不可少**: 429 是保护机制，不是 Bug
