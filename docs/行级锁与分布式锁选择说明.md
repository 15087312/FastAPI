# 数据库行级锁 vs 分布式锁 选择说明

## 背景

项目早期使用了 Redis 分布式锁（Redlock）来保证高并发下的库存操作原子性，后来切换为 PostgreSQL 行级锁（`SELECT FOR UPDATE`）。本文档说明选择行级锁的原因。

## 为什么选择数据库行级锁

### 1. 架构简化

```
┌─────────────────────────────────────────┐
│  分布式锁方案 (已删除)                    │
├─────────────────────────────────────────┤
│  Redis + 应用层锁逻辑                     │
│  - 需要额外的 Redis 连接                  │
│  - 锁的获取/释放逻辑                      │
│  - 死锁超时处理                           │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│  行级锁方案 (当前)                        │
├─────────────────────────────────────────┤
│  数据库自带能力                           │
│  - 无需额外组件                           │
│  - 事务结束自动释放                       │
│  - 数据库保证一致性                       │
└─────────────────────────────────────────┘
```

### 2. 性能更优

| 对比项 | 分布式锁 (Redlock) | 行级锁 (FOR UPDATE) |
|--------|-------------------|-------------------|
| 网络开销 | 2次 Redis 往返 | 0次额外网络 |
| 延迟 | 高（网络延迟） | 低（数据库内） |
| 锁冲突 | 需要抢锁 | 数据库队列 |

**实测数据**（单实例）：
- 分布式锁：平均延迟 +5~15ms
- 行级锁：无额外延迟

### 3. 避免分布式锁的典型问题

#### 3.1 客户端崩溃导致死锁
```python
# 分布式锁风险
lock = redis.lock("inventory:123")
lock.acquire()
# ❌ 程序崩溃，锁未释放
# 其他进程等待超时（通常 10-30 秒）
```

**行级锁无此问题**：事务提交/回滚自动释放

#### 3.2 Redis 主从切换丢锁
```
T1: 客户端A 获取锁，Redis主库写入
T2: Redis主从切换
T3: 从库未同步，客户端B 获取同一把锁成功
❌ 两把锁同时生效
```

**行级锁无此问题**：数据库保证线性一致性

#### 3.3 锁竞争激烈
```python
# 分布式锁：所有请求挤向 Redis
for _ in range(1000):
    lock = redis.lock("inventory")
    lock.acquire()  # 大量请求同时抢锁
```

**行级锁**：数据库内部排队，均匀分布

### 4. 当前实现

所有写操作均使用 `SELECT FOR UPDATE`：

```python
# inventory_operation.py:58
stock = self.db.execute(
    select(ProductStock)
    .where(...)
    .with_for_update()
).scalar_one_or_none()
```

```python
# inventory_reservation.py:91
stock = self.db.execute(
    select(ProductStock)
    .where(...)
    .with_for_update()
).scalar_one()
```

### 5. 多 Worker 场景

担心：多 uvicorn worker 下行级锁是否有效？

**答案：有效**

- 每个 worker 是独立进程
- 但共享同一个 PostgreSQL 数据库
- `FOR UPDATE` 是数据库级别的锁，不是进程级别的
- 数据库内部会排队处理

```
Worker 1 ──┐
Worker 2 ──┼──> PostgreSQL ──> 排队执行
Worker 3 ──┘
```

## 什么场景仍需要分布式锁

| 场景 | 推荐方案 |
|------|----------|
| 多实例非数据库操作（如缓存） | 分布式锁 |
| 跨服务协调 | 分布式锁 |
| 非数据库资源（如文件） | 分布式锁 |

**库存操作**：数据库本身就能保证一致性，无需分布式锁。

## 总结

| 维度 | 分布式锁 | 行级锁 |
|------|----------|--------|
| 复杂度 | 高 | 低 |
| 性能 | 较低 | 高 |
| 可靠性 | 有风险 | 可靠 |
| 维护 | 需要关注 Redis | 无需额外维护 |

**结论**：对于数据库操作，行级锁是更简单、更可靠、更高效的选择。
