# 高并发配置指南

## 📋 问题诊断

### 原因分析
你的 FastAPI 服务器使用**单进程 uvicorn 模式**运行，无法处理 1000 并发请求。

**主要问题：**
- ❌ 单个 uvicorn worker 只能使用一个 CPU 核心
- ❌ Python GIL（全局解释器锁）限制了并发性能
- ❌ 没有利用多核 CPU 的优势

## ✅ 解决方案

### 1. 已实施的优化

#### 多进程 Uvicorn 配置
修改了 `app/main.py`，添加了多进程支持：

```python
import multiprocessing

# 自动计算最优 workers 数量
cpu_count = multiprocessing.cpu_count()
workers = min(cpu_count * 2 + 1, 8)  # 最多 8 个 worker

uvicorn.run(
    "app.main:app",
    host=host,
    port=port,
    workers=workers,      # 多进程模式
    loop="uvloop",        # 使用 uvloop 提升性能
    http="httptools",     # 使用 httptools 提升 HTTP 解析
    reload=settings.DEBUG,
    access_log=True,
    log_level="info"
)
```

**workers 数量计算公式：**
- **开发环境**: 1-2 个 workers
- **生产环境**: `min(CPU 核心数 × 2 + 1, 8)`
- 示例：4 核 CPU → 9 个 workers（但限制为 8 个）

### 2. 性能优化效果

**预期提升：**
- ✅ QPS 提升 4-8 倍（取决于 CPU 核心数）
- ✅ 并发连接数大幅提升
- ✅ 更好的 CPU 利用率
- ✅ 更低的响应延迟

## 🔧 配置选项

### 环境变量配置（可选）

在 `.env` 文件中添加：


## ⚙️ 高级优化

### 1. 数据库连接池优化

当前配置（`app/db/session.py`）：
```python
engine = create_engine(
    settings.database_url,
    pool_size=10,           # 连接池大小
    max_overflow=20,        # 最大溢出连接数
    pool_pre_ping=True,     # 自动检测失效连接
    pool_recycle=1800,      # 30 分钟回收连接
)
```

**建议：**
- `pool_size` = workers 数量 × 2
- `max_overflow` = pool_size × 2

### 2. Redis 连接优化

确保 Redis 也支持高并发：

```bash
# Docker 环境 Redis 配置
redis:
  command: redis-server --maxclients 10000 --appendonly yes
```

### 3. 系统级优化

#### Linux 系统
```bash
# 增加文件描述符限制
ulimit -n 65535

# 增加 TCP 连接队列
sysctl -w net.core.somaxconn=65535
sysctl -w net.ipv4.tcp_max_syn_backlog=65535
```

#### Windows 系统
无需特殊配置，但可以：
- 增加虚拟内存
- 关闭不必要的后台程序
- 使用 WSL2 获得更好性能

## 📈 监控和调优

### 关键指标监控

1. **CPU 使用率**
   ```bash
   # Linux
   top -p $(pgrep -f 'python.*main.py')
   
   # Windows PowerShell
   Get-Process python | Select-Object CPU,WorkingSet
   ```

2. **内存使用**
   ```bash
   # Linux
   ps aux | grep python
   
   # Windows
   tasklist /FI "IMAGENAME eq python.exe" /V
   ```

3. **连接数**
   ```bash
   # 查看端口连接数
   netstat -an | findstr :8000
   ```

### 调优建议

**如果 CPU 使用率低 (< 50%)：**
- 增加 workers 数量
- 检查是否有 I/O 瓶颈

**如果内存使用过高：**
- 减少 workers 数量
- 检查内存泄漏
- 优化数据库查询

**如果响应时间过长：**
- 检查数据库性能
- 增加 Redis 缓存
- 优化代码逻辑

## 🎯 最佳实践

### 1. 生产环境部署

```bash
# 使用 Gunicorn + Uvicorn workers（可选，更稳定）
pip install gunicorn

# 启动命令
gunicorn -k uvicorn.workers.UvicornWorker \
         --bind 0.0.0.0:8000 \
         --workers 4 \
         --worker-connections 1000 \
         --timeout 120 \
         app.main:app
```

### 2. 负载均衡

对于超高并发（10000+ QPS），建议使用 Nginx 做负载均衡：

```nginx
upstream fastapi_app {
    server 127.0.0.1:8000;
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}

server {
    listen 80;
    
    location / {
        proxy_pass http://fastapi_app;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # 超时设置
        proxy_connect_timeout 60s;
        proxy_send_timeout 120s;
        proxy_read_timeout 120s;
    }
}
```

## 🔍 故障排查

### 常见问题

**Q1: 启动后只有一个进程？**
- 检查是否直接运行 `python app/main.py`
- 不要使用 `--reload` 参数（会与 workers 冲突）

**Q2: Workers 数量不生效？**
- 检查 `DEBUG=True` 时 `reload=True` 会禁用 workers
- 生产环境设置 `DEBUG=False`

**Q3: 数据库连接不足？**
- 增加 `pool_size` 和 `max_overflow`
- 减少 workers 数量

**Q4: Redis 连接超时？**
- 增加 Redis 的 `maxclients`
- 检查 Redis 服务器性能
